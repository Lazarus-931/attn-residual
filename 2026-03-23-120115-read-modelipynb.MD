⏺ The simplest fix that keeps things working: wrap it in a Flax-native list
  container.

  Option A — Use a Python list but register properly:
  Not possible in Flax. Unlike PyTorch's nn.ModuleList, Flax doesn't auto-track
  plain Python lists of submodules in setup(). They'll work at call time but
  params won't be nested/named correctly.

  Option B — Assign each expert as a named attribute:
  for i in range(self.n_routed_experts):
      setattr(self, f'expert_{i}', Expert(self.dim, self.inter_dim))
  Works, but ugly with 256 experts and still sequential.

  Option C (recommended) — Single parameterized expert + vmap:
  Define one Expert with params shaped for a single expert. Stack all expert
  params into a single pytree with a leading n_experts axis. Use vmap over the
  expert axis to apply all experts in parallel. This is the idiomatic JAX
  approach — no Python loop, fully vectorized, and Flax tracks one set of
  stacked params cleanly.

  Roughly:
  # In setup: one expert whose params get vmapped
  self.expert = Expert(self.dim, self.inter_dim)

  # In __call__: vmap over expert params
  # expert_params has shape (n_experts, ...) for each param leaf
  batched_expert = nn.vmap(Expert, variable_axes={'params': 0},
  split_rngs={'params': True},
                           in_axes=None, out_axes=0,
  axis_size=self.n_routed_experts)

  This gives you (n_experts, n_tokens, dim) output in one shot, then you
  index/weight by the gate output.

❯ The input token $x_t$ is first linearly projected by a head-specific weight   
matrix $\mathbf{W}^h_{q/k}$, producing a head-specific representation — this is 
 the standard QKV projection you'd see in any transformer, just done per head.  
That projected vector is then passed through a **ShortConv**, a small depthwise 
 convolution over a local window of recent tokens, which injects a lightweight  
form of local temporal context so the query/key isn't purely position-agnostic. 
 The convolution output is then passed through **Swish** ($x \cdot \sigma(x)$),

❯ Alrigt, how can we use jax's jit and pallas or tricks for speed on single

⏺ Here are the concrete opportunities, ordered by impact:

  1. jax.jit the forward + backward pass
  Right now nothing is jitted. Wrapping the train step (forward + loss + grad +
  optimizer) in a single @jax.jit is the single biggest win — it fuses the
  entire computation graph into one XLA program. This is table stakes.

  2. Replace the MoE expert loop with Pallas
  Even with vmap, all 256 experts run on all tokens. On TPU v6, a Pallas kernel
  can do the scatter-gather natively — group tokens by expert assignment, run
  each expert only on its assigned tokens in a fused megakernel, scatter results
   back. This avoids the O(n_experts × n_tokens) waste of the current approach.

  3. Pallas FlashAttention for the MLA path
  The MLA softmax attention (einsum → softmax → einsum) is the classic target
  for flash attention. jax.experimental.pallas.ops.tpu.flash_attention exists
  and is tuned for TPU v6's HBM hierarchy. Drop-in replacement for the manual
  einsum chain in multihead_attn.__call__.

  4. Fuse the KDA ShortConv + Swish + L2Norm into a single Pallas kernel
  The Q/K pipeline does: Dense → ShortConv → swish → normalize. The ShortConv
  currently uses a Python for-loop over kernel taps — XLA will unroll it, but a
  Pallas kernel can do the depthwise conv + activation + norm in one pass
  without materializing intermediates.

  5. jax.checkpoint (gradient rematerialization)
  With 300M params and long sequences, you'll OOM on the backward pass.
  jax.checkpoint on each BottomBlock/TopBlock trades recomputation for memory —
  critical for fitting 2048 context on a single device.

  6. bf16 everywhere, f32 only for loss + norms
  You're already casting to bf16 in KDA, but the MLA path and MoE path don't.
  TPU v6 MXU is 2x faster on bf16. Use jmp (mixed precision) or manual casts —
  keep RMSNorm and softmax in f32, everything else in bf16.

  7. scan over layer groups instead of Python loop
  The Transformer forward loops over layers in Python. If BottomBlocks share the
   same structure, nn.scan over the 3 BottomBlocks per group means XLA compiles
  one block and tiles it, slashing compile time from O(n_layers) to O(1).

  8. Shard the MoE expert params across TPU cores
  Even on a single v6 chip, there are multiple cores. jax.sharding with
  NamedSharding can split the expert vmap axis across cores so each core holds a
   subset of expert params.

  Priority order for implementation:
  1. @jax.jit the train step (free, massive)
  2. jax.checkpoint per block (free, enables longer sequences)
  3. bf16 mixed precision (near-free, 2x MXU throughput)
  4. nn.scan over layers (moderate effort, faster compilation)
  5. Pallas flash attention for MLA (moderate, big win on long seqs)
  6. Pallas MoE kernel (high effort, biggest throughput win)
  7. Fused ShortConv kernel (moderate, moderate win)

⏺ Yes. jax.jit works fine with nn.compact — the jit boundary goes around the

  @jax.jit

  jit needs. Whether the module internally uses @nn.compact or setup() doesn't

  model.apply() is just a stateless function call.

⏺ For a Kimi Linear + AttnRes experiment on single TPU v6, here's what's worth
  logging beyond training loss:

  Validation metrics:
  - Val loss (most important — the whole experiment question)
  - Perplexity (exp(val_loss))

  Learning dynamics:
  - Gradient norm (global, before clipping)
  - Param norm (detect divergence early)
  - Learning rate (verify schedule is correct)
  - Muon momentum buffer norm (sanity check the optimizer)

  AttnRes-specific (unique to this experiment):
  - alpha weights from FullAttnResiduals / BlockAttentionResidual — log the
  softmax distribution over layers per block. This is the key diagnostic: do
  deeper layers attend to earlier layers? Does the distribution sharpen or
  flatten during training? This is what the paper visualizes.
  - Entropy of the alpha distribution — scalar summary of whether AttnRes is
  using many layers (high entropy) or collapsing to one (low)

  MoE-specific:
  - Expert load balance — how many tokens each expert gets. The
  e_score_correction_bias should be drifting to balance this
  - Router entropy — are tokens being spread across experts or collapsing?
  - Fraction of tokens routed to shared expert vs routed experts

  Throughput:
  - Tokens/sec
  - Step time (wall clock)
  - MXU utilization (via JAX profiler, not in-code)

  Memory:
  - Peak HBM usage (via jax.local_devices()[0].memory_stats())

  Priority for your experiment: val loss, perplexity, gradient norm, learning
  rate, and the AttnRes alpha distributions. The last one is the novel
  contribution — it tells you whether attention residuals are actually learning
  meaningful cross-layer dependencies vs degenerating to standard residuals.
